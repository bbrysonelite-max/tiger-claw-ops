# Builder Agent Blueprint (OpenClaw + Code Assistant)

## Purpose

The **Builder Agent** is a non-human, non-interactive worker that:

- Watches the `/tasks/` directory for new task definitions.
- For each new task JSON, orchestrates code changes using a code assistant (e.g. Claude Code or Cursor).
- Runs tests/commands defined in the task.
- Writes machine-readable status updates to `/status/` that Agent Zero and dashboards can consume.

This removes the human from the copy/paste loop and lets Agent Zero act as architect/orchestrator by writing tasks to disk.

---

## Directories

- **Base**: `/usr/projects/tiger_bot_multiagent`
  - `docs/blueprints/` – long-form specs and architecture docs.
  - `tasks/` – JSON tasks for the Builder Agent.
  - `status/` – JSON status files generated by the Builder Agent.
  - `task.schema.json` – JSON Schema describing the structure of a task.
  - `status.schema.json` – JSON Schema describing the structure of a status file.

---

## Task Lifecycle

1. **Creation** (Agent Zero)
   - Agent Zero writes a new task JSON file into `/usr/projects/tiger_bot_multiagent/tasks/`.
   - File name convention: `<timestamp>-<slug>.task.json`, e.g. `2026-02-08-analytics-endpoint.task.json`.
   - The JSON MUST conform to `task.schema.json`.

2. **Pickup** (Builder Agent)
   - Builder Agent polls the `tasks/` directory on an interval (e.g. every 10s).
   - For each `*.task.json` not yet processed:
     - Reads the JSON.
     - Validates against `task.schema.json` (optional but recommended).
     - Sets initial status:
       - Writes `/status/<task-id>.status.json` with `{ taskId, status: "queued" }`.

3. **Execution** (Builder Agent + Code Assistant)

For each task step in `task.steps`:

- Update status for the step to `running`.
- Depending on `kind`:
  - `edit_files` / `create_files` / `refactor` / `documentation`:
    - Invoke the code assistant (Claude Code / Cursor) with:
      - `repoPath` from the task.
      - `targetFiles` and `instruction`.
      - Any `contextFiles` paths for additional context.
  - `run_command` / `write_tests`:
    - Execute each shell command listed in `commands` inside `repoPath`.
    - Capture exit code and logs.
- Update per-step status to `success` or `failed`.

The Builder Agent does **not** invent behavior. It follows `steps` exactly.

4. **Testing**

If the task specifies `steps` with `kind: "run_command"` for tests (e.g. `npm test`, `pytest`):

- Run them after code modifications.
- Record:
  - `tests.command`
  - `tests.exitCode`
  - `tests.passed` (exit code == 0)
  - `tests.logPath` (optional path to a log file)

5. **Completion**

- After all steps are processed:
  - If all steps `success` and tests (if any) pass → `status: "success"`.
  - If any step fails or tests fail → `status: "failed"` (or `"partial"` if some steps succeeded).
- Update `summary` with a short human-readable description.
- Update `updatedAt` with current ISO timestamp.

---

## Status File Convention

- File name: `<task-id>.status.json`, where `task-id` is the `id` field from the task JSON.
- Location: `/usr/projects/tiger_bot_multiagent/status/`.
- MUST conform to `status.schema.json`.

Example minimal success status:

```json
{
  "taskId": "2026-02-08-hive-learnings-page",
  "status": "success",
  "summary": "Implemented Hive Learnings leaderboard UI and wired to /ai-crm/hive/leaderboard.",
  "stepStatuses": [
    { "stepId": "step-1", "status": "success" },
    { "stepId": "step-2", "status": "success" }
  ],
  "tests": {
    "command": "npm test",
    "exitCode": 0,
    "passed": true
  },
  "logs": [
    "Applied changes to app/dashboard/hive/page.tsx",
    "Tests: 23 passed, 0 failed"
  ],
  "updatedAt": "2026-02-08T23:00:00Z"
}
```

## Integration Points

- **Agent Zero**
  - Writes tasks to `/usr/projects/tiger_bot_multiagent/tasks/`.
  - Reads statuses from `/usr/projects/tiger_bot_multiagent/status/`.
  - Uses schemas to validate and reason about work state.

- **OpenClaw / Tiger Claw**
  - One OpenClaw agent is configured as the Builder Agent.
  - It knows:
    - The task directory path.
    - How to talk to the code assistant (e.g. local HTTP endpoint, CLI wrapper).
    - How to run commands inside the repo.

- **Code Assistant (Claude Code, Cursor)**
  - Receives highly-structured prompts from the Builder Agent.
  - Operates directly on the repo at `repoPath`.
  - Does not need to know about task/status schemas, only about specific file-level instructions.

---

This blueprint is the contract: as long as the Builder Agent and Agent Zero both respect these schemas and paths, the human can remain out of the copy/paste loop while still retaining full visibility and control.
